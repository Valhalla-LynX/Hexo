[设计模式][中文]管道(管线)设计模式

发表于2022-03-21|更新于2023-09-19|设计模式
|阅读量:

摘要

Doc: 什么是管道设计模式. [English] | [中文]

管道模式是责任链模式的常见变体之一。 在责任链模式下，链上只有一个处理器来处理数据，而在流水线模式下，多个处理器将处理数据。

引用维基百科：

In software engineering, a pipeline consists of a chain of processing elements (processes, threads, coroutines, functions, etc.), arranged so that the output of each element is the input of the next; the name is by analogy to a physical pipeline.

介绍

目的

处理输入信息以供下一阶段使用。 减少连续任务功能的耦合。

解释

图像与现实世界管道的类比。 数据在管道中流动，管道上有多个阀门。 阀门可以看作是处理数据的任务节点。 阀门对数据进行操作并以相同的格式返回数据以供下一个阀门使用。 这样，管道和阀门就可以看作是定义了任务方法的类，可以以链表的方式相互组合。 多条数据流通过管道代表并发计算。

流水线模式是一条流水线，其中部分结果从一个阶段传递到另一个阶段。

优点

  • 减少耦合。 类似的任务功能被抽象成任务基类，任务只需要实现主要的抽象方法即可。
  • 简化数据对象。 数据对象不需要刻意适应任务功能。
  • 任务阀门可以动态组合添加、删除或更改顺序。ask 阀门可以动态组合或按顺序更改。
  • 当代码中存在大量异常时，可以改善臃肿的 try-catch 块。
  • 兼容并发计算。

注意

对数据的操作可以分为两类。 第一类不改变数据主体，只是改变部分信息，部分属性信息可能不会被修改。 另一种是数据主体会发生变化，操作后数据会发生很大变化。 前者可视为上下文，修改为上下内部信息； 而后者则需要以流水线方式构造输入输出对象，输入输出对象可能存在较大差异或无法修改。 后一种对象的过渡构造在流水线模式下可能会产生大量的时空开销。

在使用流水线模式时，开发者需要明确需要哪些输入输出信息，以免项目因为使用这种模式而更加混乱。
