[实时数仓][Flink][DataStream API]实时数仓

发表于2022-04-28|更新于2023-09-19|大数据
|阅读量:

摘要

早期实时计算需求量小，多流ETL的直接关系少。伴随着业务增长，每个需求做一个ETL业务流的模式暴露出来的问题也越来越多。

  • 数据指标增多，代码耦合问题严重。
  • 需求资源增加，成本增加。
  • 系统数据链路监控困难。

为了更好的治理数据，提出了数据分层的概念。
实时数仓通过数据分层架构，缓解数据存储、检索、聚合和计算等操作的压力。
有关实时数仓架构的详情不在本文中介绍。
本文主要关注通过FLink DataStream API实现实时数仓的基本Demo。
其逻辑可以看作应用查询时才进行的复杂聚合，转化为数据生成时就聚合完成的表，用空间换时间。

案例技术架构

环境

  • Flink 1.14.4-scala2.11-java8
  • Kafka 3.1.0-scala2.13
  • MySQL8
  • HBase/Kudo

部署

本文部署使用三台机器组成Kubernetes（K8s）集群，使用Flink Session on Kubernetes的方式部署。
关于Kafka和数据库使用单机模式，分布式集群的部署不再介绍。

事项

其次，修改hostname和hosts，同步三台机器的host。
最后，从官网获取Flink、Kafka、MySQL等包，解压到工作空间。

Flink Kafka

CDC会在每次任务提交，DAG生成完毕，任务开始执行时，将mysql bin log全部推出来。

  • OffsetResetStrategy.EARLIEST 策略会将启动前的log数据同步到kafka log。
  • OffsetResetStrategy.LATEST 策略仅会将启动后，新的log数据同步到kafka log。

DIM

Flink Connector

默认的MySQL Sink使用的preparedstatement需要预先确定sql语句中的table表名。而现在实现中的Kafka拿到的流数据中包括多个表信息。因此，需要自定义MySQL Sink。

DWD

Flink Operator

在Kafka Source这种监听了两个topic到一个流中，在process中将其分为连个流，再使用window join的方式，将数据聚合为新的流。

（DWM DWS APP）

同理，该层对聚合的新数据流和DIM层中的商品关联聚合就得到了推荐商品表，Sink到MySQL中就可以供OLAP等平台应用使用了。
